# Dungeon Clash — Поэтапный план реализации

**Основано на:** `TECH_STACK.md`, `dungeon_clash_GDD.md` v0.3
**Статус:** предложение к обсуждению

Фазы упорядочены **по зависимостям, а не по календарю** (оценки в неделях зависят от размера команды — добавлю, если скажешь). Каждая фаза заканчивается **проверяемым критерием готовности** и расширяет тестовый набор. Сквозной принцип: как можно раньше получить **тонкий вертикальный срез** (ядро → стратегия → CLI), затем углублять.

---

## 0. Принципы, которые держим во всех фазах

1. **Граница ядро/адаптеры священна.** `core` не импортирует Textual/Rich/SQLite/Gymnasium/FastAPI. Проверяется `import-linter` в CI с первого коммита (§9, §10 tech-stack).
2. **Ядро детерминированное и headless.** Любое поведение воспроизводимо по `seed`. Это фундамент для логов, пассивного режима, RL и анти-чита.
3. **Контент — данные, не код.** Враги/реликвии/оружие/события в TOML, валидируются Pydantic.
4. **Тестируем вместе с кодом, не после.** Каждая фаза вводит свой слой тестов (см. главу «Тестирование»).
5. **Вертикальный срез вперёд глубины.** Сначала «бот проходит один бой end-to-end», потом богатство систем.

## Фаза 0 — Каркас и границы

**Цель.** Скелет, в котором архитектурные правила выполняются автоматически.

**Поставка.**
- Структура пакета:
  ```
  dungeon_clash/
    core/         # детерминированное ядро: state, combat, systems, events, rng
    content/      # загрузчик + TOML-данные + Pydantic-схемы
    strategy/     # протокол Strategy, песочница, хелперы действий
    adapters/
      render/     # Rich/Textual
      persist/    # SQLite
      rlenv/       # Gymnasium/PettingZoo
    cli/          # Typer
  tests/
  ```
- `pyproject.toml` (Hatchling), окружение через `uv`, entry point `dungeon = dungeon_clash.cli:app`.
- `ruff`, `mypy --strict` на `core`, `pre-commit`, GitHub Actions.
- **`importlinter`-контракты**: `core` ⇏ {adapters, cli, textual, rich, sqlite3, gymnasium, fastapi}.

**Критерий готовности.** Пустой CI зелёный: линт, типы, контракты границ, `pytest` (хотя бы smoke).

**Тесты фазы.** Контрактные (import-linter), smoke-импорт пакета.

## Фаза 1 — Детерминированное боевое ядро

**Цель.** Перенести и формализовать боёвку прототипа (`resolve()`, зоны, AI) в чистое ядро.

**Поставка.**
- Pydantic-модели: `GameState`, `Combatant`, `Enemy`, `Action`, `Event` (+ подтипы: `HitEvent`, `MissEvent`, `BlockEvent`, `StressEvent`, `DeathEvent`, `StrategyErrorEvent`…).
- `rng.py`: явный seeded-генератор (фиксированный алгоритм, **без** глобального `random`).
- `combat.step(state, action, rng) -> (state, events)`: формула урона (GDD 7.2), хитчансы зон (7.1), зональная защита, enemy bias-AI (7.3).
- **Решение про числа:** урон/блок в **целочисленной/fixed-point** арифметике, чтобы детерминизм был bit-identical между платформами (важно для replay-анти-чита).

**Критерий готовности.** Один и тот же seed + последовательность действий → побитово идентичный поток событий. Инварианты держатся под фаззингом.

**Тесты фазы.** Unit (формула урона на границах), property/Hypothesis (инварианты), replay-детерминизм, stateful-фаззинг (см. главу тестирования, T2/T4/T5).

## Фаза 2 — Стратегии и песочница v1

**Цель.** «Strategy as code»: функция игрока управляет боем.

**Поставка.**
- `Strategy`-протокол `decide(state) -> Action`; хелперы `attack/defend/flee/extract/rest/cast/use_item`.
- **Frozen-снимок `state`** для стратегии (нельзя мутировать ядро).
- In-process исполнение с ловлей исключений → `StrategyErrorEvent` с трейсбэком, ход пропускается (GDD 4.2 — by design).
- `ScriptedBot` (эвристика) для тестов и болванчиков.

**Критерий готовности.** Скриптовый бот проводит полный бой; кривая стратегия (исключение/невалидное действие/таймаут) не роняет движок, а логируется.

**Тесты фазы.** Стратегические сбои (T7), интеграция «бот ⨯ ядро» (T8).

---

## Фаза 3 — Персистентность и пассивный режим

**Цель.** Состояние переживает сессии; «играет, пока ты работаешь» — без демона.

**Поставка.**
- SQLite-схема: append-only `event_log`, снапшот прогона (versioned blob), мета-прогресс, `last_checked_at`, `seed`.
- **Lazy catch-up**: при `dungeon status` ядро детерминированно досчитывает прошедшие тики и аппендит лог (GDD 4.3 «не наказывать за отсутствие»).
- CLI (Typer): `dungeon status`, `dungeon log [--last N] [--stats]`.

**Критерий готовности.** Закрыть процесс, подождать, открыть `dungeon status` — корректно досчитанный прогресс. Lazy-sim даёт **тот же** результат, что пошаговая симуляция.

**Тесты фазы.** Эквивалентность lazy/пошаговой симуляции (T8), round-trip сериализации (T2), CLI (T9), агрегаты `--stats` (T1).

## Фаза 4 — Активный TUI и рендер-адаптер

**Цель.** `dungeon play` + бесшовное переключение режимов.

**Поставка.**
- Рендер-адаптер: `render(event) -> Rich renderable` (единый для пассивного вывода и Textual).
- Textual-`App`: боевой экран (спрайты, HP-бары, таблица зон, `RichLog`), биндинги H/T/L.
- Переключение passive↔active: загрузка/выгрузка снапшота в точку выхода (GDD 4.3).
- Fallback без TTY/цвета (`NO_COLOR`, `--no-color`).

**Критерий готовности.** Сыграть бой руками, выйти, пассивная стратегия продолжает с того же состояния.

**Тесты фазы.** Снапшот-рендер (T3), TUI через `run_test()`/Pilot (T10), тест переключения режимов (T8).

## Фаза 5 — Полная структура прогона

**Цель.** Из «одного боя» — в «прогон по этажам».

**Поставка.**
- Этажи, ветвящаяся карта комнат, типы комнат (GDD 6.2), навигация (`map_choice`).
- Система стресса с порогами и эффектами (GDD 8), читаемая в UI.
- Ресурсы (золото/руда/материалы/кристаллы), штрафы смерти (10.3).
- Push-your-luck: `extract`/`continue`, множители (15.4).

**Критерий готовности.** Бот проходит многоэтажный прогон, корректно отрабатывают стресс-пороги, extract и штрафы смерти.

**Тесты фазы.** Пороги стресса (property, T2), стат-симуляции баланса (T12), интеграция полного прогона (T8).

## Фаза 6 — Глубина контента

**Цель.** Наполнить системами и контентом из GDD.

**Поставка.**
- Контент-пайплайн: TOML + Pydantic + загрузчик + **референс-валидация** (рецепты ссылаются на существующие материалы, дропы — на предметы).
- Полный бестиарий с уникальными механиками (GDD 12), боссы с фазами.
- Реликвии (14), экипировка/крафт (11), Hall of Upgrades (15.2), события (13), магия и заклинания (9).

**Критерий готовности.** Весь контент GDD загружается, валиден, механики срабатывают в боях (по тесту на механику).

**Тесты фазы.** Валидация контента + референс-целостность (T6), по unit-тесту на каждую особую механику врага/реликвии (T1), снапшот-логи эталонных боёв (T3), баланс-симуляции (T12).

## Фаза 7 — RL-среда

**Цель.** Ядро как обучающая среда; ML-policy в том же `Strategy`-слоте.

**Поставка.**
- Gymnasium-адаптер: `encode(state)->obs`, `MultiDiscrete` action space, `action_mask(state)`, подключаемый reward от событий.
- PettingZoo-обёртка (self-play/мульти-враг).
- Векторизация (`AsyncVectorEnv`); опц. extras с примерами тренеров (SB3/CleanRL).

**Критерий готовности.** `gymnasium.utils.env_checker.check_env` и `pettingzoo.test.api_test` зелёные; baseline (MaskablePPO) на floor 1–2 обучается лучше случайного.

**Тесты фазы.** Env-checkers (T11), корректность маски, знак/границы reward, воспроизводимость по seed, консистентность векторных env, перф-throughput (T13).

## Фаза 8 — MMO-слой (v3+)

**Цель.** Общий мир, лидерборды, исполнение чужих стратегий.

**Поставка.**
- FastAPI + PostgreSQL; те же Pydantic-модели.
- **Replay-анти-чит**: сервер реиграет (seed + стратегия) тем же ядром, сверяет результат.
- Песочница чужих стратегий: subprocess+rlimit → WASM (Pyodide)/Firecracker.
- Лидерборды, async co-op, гильдии; market-data cron (v4); контент-PR + валидация в CI.

**Критерий готовности.** Подменённый прогон отклоняется replay-проверкой; чужая стратегия исполняется изолированно (нет сети/ФС, есть таймаут).

**Тесты фазы.** Серверные (TestClient), replay-валидация, DB-интеграция (testcontainers), безопасность песочницы (T7-расширенный).

# Тестирование — подробно

Тестирование организовано **пирамидой**: много дешёвых детерминированных юнит/property-тестов на ядре, меньше интеграционных, ещё меньше end-to-end и «мягких» статистических. Фундамент — **детерминизм ядра**: без скрытого `random`, без I/O в `core`, всё за явным `seed`. Это делает почти всё остальное тестируемым честно.

## Категории тестов

### T1 — Unit (pytest)
Чистые функции ядра: формула урона (7.2) на граничных значениях, хитчансы, зональная защита, переходы порогов стресса (8.1), штрафы смерти (10.3), reward-функции, `encode`/`action_mask`, агрегаты `--stats`. По **отдельному тесту на каждую уникальную механику** врага/реликвии (Reassemble, Regeneration, Blood Frenzy, Mirror Block, Pack Fury, Holy Water +25% и т.д.) — это места, где баги тихие.
*Инструменты:* `pytest`, `pytest-cov`. **Гейт покрытия на `core` ≥ 90%** (на адаптерах ниже).

### T2 — Property-based (Hypothesis)
Инварианты, которые должны держаться при любых входах:
- HP/стресс/ресурсы всегда в допустимых границах, ресурс не уходит в минус;
- упорядоченность порогов стресса (Calm < Rattled < Panicking < Breaking) при любых модификаторах шлема;
- монотонность урона по множителю атаки;
- **round-trip сериализации**: `state == load(dump(state))`;
- маскированные действия **никогда** не применяются;
- штрафы смерти строго в заявленных долях (Gold −50%, Ore −25%, мета −0%).
*Инструменты:* `hypothesis`.

### T3 — Snapshot / golden (syrupy + textual-snapshot)
- Рендер: `rich.Console(record=True)` → текстовый снапшот строк боя; изменения видны в diff.
- TUI: `pytest-textual-snapshot` для экранов.
- **Golden-логи**: полный прогон с фиксированным seed и эталонной стратегией → снапшот потока событий. Любое непреднамеренное изменение правил мгновенно ловится.
*Инструменты:* `syrupy`, `pytest-textual-snapshot`.

### T4 — Детерминизм и replay (краеугольный)
Тот же seed + те же действия → **побитово** идентичный поток событий и финальный `state`. Это одновременно: гарантия воспроизводимости логов (GDD 4.2 «трассируемые смерти»), корректности lazy-sim и **основа MMO-анти-чита** (Фаза 8). Отдельно — **кросс-платформенный** прогон в CI-матрице (Linux/macOS/Windows, Python 3.11/3.12/3.13): хэш golden-лога должен совпадать везде (отсюда требование fixed-point арифметики из Фазы 1).

### T5 — Stateful-фаззинг (Hypothesis RuleBasedStateMachine)
Моделируем игру как state-machine: Hypothesis генерирует длинные случайные последовательности легальных действий и после каждого шага проверяет инварианты из T2. Для систем-тяжёлой игры это лучший способ найти редкие каскады (стресс ⨯ реликвия ⨯ механика врага), которые ручные тесты пропускают. Найденный контрпример Hypothesis минимизирует автоматически.

### T6 — Контент и референс-целостность
Каждый TOML-файл валиден против Pydantic-схемы; рецепты крафта ссылаются на существующие материалы; дропы — на существующие предметы; у каждого врага есть спрайт/ассет-ID. Запускается в CI и **гейтит community-PR** (GDD v4) — контрибьютор не сможет влить ломаный контент.

### T7 — Песочница стратегий (безопасность)
Кривые и враждебные стратегии: бесконечный цикл → прерывается по таймауту; исключение → логируется, ход пропускается; невалидное/недоступное действие → отклоняется; (для изолированных тиров) попытка доступа к сети/ФС/импорту → блокируется. В Фазе 8 расширяется до полноценных тестов изоляции (WASM/subprocess).

### T8 — Интеграция / вертикальные срезы
- скриптовый бот проходит полный многоэтажный прогон end-to-end;
- **эквивалентность lazy-sim ↔ пошаговой** симуляции (одинаковый результат);
- переключение режимов passive→active→passive сохраняет состояние;
- цикл «смерть → штрафы → мета-прогресс сохранён».

### T9 — CLI (Typer CliRunner)
`dungeon status/log/play`, опции `--last N`, `--stats`, коды возврата, поведение при пайпинге (`| grep`), `--no-color`/`NO_COLOR`.

### T10 — TUI (Textual Pilot)
`App.run_test()`: эмуляция нажатий H/T/L, проверка состояния экрана, корректность HP-баров/таблицы зон, обработка выхода.

### T11 — RL-среда
`gymnasium.utils.env_checker.check_env`, `pettingzoo.test.api_test`; obs ∈ space; корректность `action_mask` (маска ⇔ реальная легальность); знак и границы reward; терминация эпизода; воспроизводимость по seed; консистентность векторных env между собой.

### T12 — Баланс / статистические симуляции (мягкие)
Тысячи прогонов эталонными стратегиями → распределения: винрейт по этажам в ожидаемом коридоре, средняя длина прогона, отсутствие невыносимых/тривиальных врагов. Это **мягкие** тесты с допусками; гоняются **ночью**, не на каждый коммит. Ловят регрессии баланса и заодно дают данные для тюнинга.

### T13 — Производительность (pytest-benchmark)
Бенчмарк throughput `step()` (шагов/сек) с гейтом «не ниже порога» — критично для RL (Фаза 7). Ночной прогон ловит перф-регрессии. При просадке — сигнал к Тиеру 2/3 throughput-стратегии (§10 tech-stack).

### T14 — Мутационное тестирование (периодически)
`mutmut`/`cosmic-ray` по `core`: проверяет, что тесты реально ловят изменения логики, а не просто исполняют код. Гоняется не на каждый коммит, а перед релизами — это аудит качества самого тест-набора на самой критичной части.

### T15 — Серверные (Фаза 8)
FastAPI `TestClient`; replay-валидация (подменённый прогон отклоняется); DB-интеграция на реальном Postgres через `testcontainers`; корректность лидербордов; авторизация.
