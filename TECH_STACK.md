## 0. TL;DR

| Слой | Выбор | Почему |
|------|-------|--------|
| Язык ядра | **Python 3.11+** | Стратегия-как-код = Python (раздел 5 GDD, неоспоримо). Прототип уже на Python, аудитория — разработчики. |
| Активный TUI (`dungeon play`) | **Textual** | Event-driven модель идеально ложится на «ничего не происходит, пока ты не примешь решение». |
| Рендер логов/статуса | **Rich** | Стилизованные таблицы/панели/ASCII под скриншот для `dungeon status` / `dungeon log`. |
| CLI | **Typer** (на Click) | Подкоманды `status/log/play`, опции `--last 5`, `--stats` через type hints. |
| Ядро симуляции | **Чистый Python, без I/O, детерминированное** | Seeded RNG → воспроизводимые логи, ленивая симуляция, серверный replay для MMO. |
| Модель данных / контент | **Pydantic v2 + TOML-файлы** | Валидация контента, бесшовный путь к «контенту через PR» (v4) и общим моделям клиент/сервер. |
| Персистентность | **SQLite (stdlib `sqlite3`)** | Append-only лог, агрегаты для `--stats`, мета-прогресс. Один файл, ноль конфигурации. |
| Пассивный режим | **Ленивая offline-симуляция, НЕ демон** | «Никогда не наказывать за отсутствие», работает после ребута, не ест ресурсы. |
| Песочница стратегий | v1: in-process; v2+: **WASM (Pyodide) / subprocess** | Свой код — доверенный; чужой (sharing/MMO) — нет. Главный риск безопасности. |
| Тесты | **pytest + Hypothesis** | Детерминированное ядро тривиально тестировать; property-тесты для инвариантов боя. |
| Упаковка | **pyproject + Hatchling + uv, ставится через pipx** | `pipx install dungeon-clash` → команда `dungeon` в PATH. |
| Dev-тулинг | **ruff + mypy + pre-commit + GitHub Actions** | Контракт `state`-API, от которого зависят стратегии игроков, нужно держать типизированным. |
| RL / ML-агенты | **Gymnasium + PettingZoo** (env), тренеры опционально | Ядро уже headless+детерминированное → это RL-среда. ML-policy = тот же `Strategy`-слот, что и человек. |
| MMO backend (v3+) | **FastAPI + PostgreSQL** | Async, Pydantic-native (те же модели), серверный replay ядра для анти-чита. |

**Главный архитектурный принцип:** строго разделить **детерминированное ядро симуляции** (без I/O и рендера) и **презентацию** (TUI/CLI/сервер). Всё остальное вытекает отсюда — в том числе возможность позже подключить улучшенную графику сменой одного адаптера рендера (§9) и использовать ядро как RL-среду для ML-агентов (§10), не трогая логику игры. Рендер, persistence, RL-env и MMO-сервер — все они адаптеры к одному ядру.

## 1. Почему Python — не обсуждается

Раздел 5 GDD делает Python-функцию игрока **основным механизмом прогрессии в середине-конце игры**. Стратегия исполняется движком, получает `state` и возвращает действия. Это значит:

- Движок должен исполнять пользовательский Python нативно, в процессе, с чистым API-неймспейсом.
- Аудитория (раздел 3) — разработчики, которым комфортно с Python и терминалом.
- Прототип уже на Python (рабочий combat-цикл, классы, бестиарий, прогрессия).

Переписывать ядро на Go/Rust/C означало бы убить ключевой пиллар. Python 3.11+ берём ради скорости, `tomllib` в stdlib и **улучшенных сообщений об ошибках** — а «дебаг своей стратегии — это часть геймплея» (раздел 4.1), так что качество трейсбэков напрямую влияет на UX.

## 2. Презентационный слой: Textual + Rich + Typer

Скриншот задаёт планку: цветной ASCII, HP-бары, таблицы зон, лог резолюции. Это всё из коробки даёт экосистема Textualize.

### Активный режим — Textual
`dungeon play` (раздел 4.2 GDD) — **строго event-driven**: «ничего не происходит, пока ты не примешь решение». Это ровно реактивная модель Textual: виджеты, async-обработчики событий, layout боевого экрана как композиция панелей. Переключение режимов «бесшовно, возврат в точку выхода» реализуется как загрузка состояния из ядра в TUI-приложение.

### Пассивный рендер — Rich
`dungeon status`, `dungeon log`, `dungeon log --last 5` — это не интерактив, а форматированный вывод: панели, таблицы, подсветка трейсбэков `[STRATEGY ERROR on turn 4: KeyError ...]`. Rich (тот же автор, что и Textual) делает это минимальными усилиями и держит единый визуальный язык.

### CLI — Typer
Подкоманды и опции из GDD (`status`, `log`, `play`, `--last N`, `--stats`) ложатся на type-hint-based CLI. Entry point `dungeon = dungeon_clash.cli:app`.

> Рассмотренные альтернативы: `curses`/`blessed` — слишком низкоуровнево; `prompt_toolkit` — мощно, но Textual выше уровнем и современнее. Click напрямую вместо Typer — допустимо, но Typer идиоматичнее для аудитории.

## 3. Ядро симуляции: детерминированное, без I/O

Это сердце архитектуры. Ядро — чистые функции:

```
step(state, action, rng) -> (new_state, events)
```

Никакого `print`, никакого ввода, никакого доступа к файлам внутри ядра. Что это даёт — закрывает сразу несколько требований GDD:

| Свойство ядра | Требование GDD |
|---------------|----------------|
| **Seeded RNG** (`random.Random(seed)` на run/floor) | «Каждая смерть логируется детально», воспроизводимость, лидерборды по эффективности стратегий (v3) — серверу нужно реиграть прогон. |
| **Один источник истины для боя** | И активный TUI, и пассивная симуляция, и серверный replay используют один код → нет рассинхрона правил. |
| **Чистота → тестируемость** | Системы сильно переплетены (стресс↔бой↔ресурсы, раздел 2). Property-тесты ловят регрессии. |
| **Snapshot состояния** | Бесшовное переключение режимов = сериализовать/десериализовать `state`. |

`state`, передаваемый в стратегию, — **frozen Pydantic-модель** с вычисляемыми свойствами (`hp_pct`, `stress_pct` из раздела 5.2). Действия (`attack/defend/flee/extract/rest/cast/use_item`) — импортируемые хелперы, возвращающие лёгкие объекты-намерения, которые ядро валидирует.

## 4. Пассивный режим: ленивая симуляция, а не демон

Критическое решение. GDD (раздел 4.1, 4.3) категорически требует: «работает корректно, даже если ты не смотрел на игру три дня», «ничего не истекает», «никогда не наказывать за отсутствие».

**Не делаем фоновый демон** — он ест ресурсы, умирает при ребуте/закрытии терминала, и провоцирует «babysitting», который GDD прямо называет топ-жалобой жанра.

**Делаем offline catch-up:** храним `last_checked_at`, текущий `state` и RNG-seed. Когда игрок запускает `dungeon status`, ядро **детерминированно досчитывает** прошедшие «тики» вперёд (например, N комнат в час, с разумным капом), исполняя стратегию игрока, и аппендит результат в лог. Бесплатно, переживает ребут, по определению «не наказывает за отсутствие». Чистый фит с детерминированным ядром из §3.

## 5. Контент как данные (Pydantic + TOML)

GDD — это огромный объём структурированного контента: бестиарий (разделы 12), реликвии (14), оружие/броня/шлемы/аксессуары (11), заклинания (9), события (13). Хардкодить это в `.py` (как сейчас в прототипе) не масштабируется.

**Контент = TOML-файлы, валидируемые Pydantic-моделями.** Почему TOML: дружелюбен к разработчикам, разрешает комментарии, читаем в diff'ах. Почему Pydantic: валидация на загрузке ловит ошибки контента рано.

Главный выигрыш — **v4 «контент через PR + голосование»** становится тривиальным: контрибьютор присылает PR с одним TOML-файлом, CI валидирует его против схемы Pydantic. Никакого кода — только данные.

> Лёгкая альтернатива: `dataclasses`. Pydantic выигрывает именно из-за сценария валидации контента и общих моделей клиент/сервер в MMO.

## 6. Персистентность: SQLite

Один файл в `~/.dungeon_clash/` (XDG-совместимо), через stdlib `sqlite3`. Покрывает:

- **Append-only event log** (`dungeon log`) — естественно для таблицы с автоинкрементом.
- **Агрегаты для `dungeon log --stats`** (avg damage dealt/taken, gold/floor, причины смертей — раздел 4.2) — это буквально SQL `GROUP BY`/`AVG`.
- **Мета-прогресс** (Hall of Upgrades, разблокированные этажи, инвентарь материалов — раздел 15) — то, что «никогда не теряется при смерти».
- **Снимок текущего прогона** — как versioned JSON/msgpack-блоб (схема через Pydantic).

SQLite: ноль конфигурации, транзакционность (важно при аппенде лога во время catch-up), один файл легко бэкапить. ORM (SQLAlchemy) для v1 — оверкилл; тонкий слой поверх `sqlite3` достаточно до появления сервера.

## 7. Песочница для стратегий — главный риск безопасности

| Этап | Чей код исполняем | Подход |
|------|-------------------|--------|
| **v1 (одиночка)** | Свой собственный | **In-process.** Это машина игрока и его код. Передаём frozen `state`, ловим исключения, логируем трейсбэк, пропускаем ход (раздел 4.2 — это by design). |
| **v2 (strategy sharing)** | Чужой | Нужна изоляция: `RestrictedPython`, либо subprocess с `rlimit`/seccomp, тайм-аутами и без сети/ФС. |
| **v3+ (MMO)** | Чужой, на сервере | **WASM (Pyodide)** или контейнер/Firecracker на стратегию. Серверный replay через то же ядро для анти-чита лидербордов. |

Закладываем чистую границу API уже в v1 (стратегия видит только `state` + хелперы действий, ничего больше), чтобы переход к песочнице не потребовал переписывать контракт.

## 8. Тесты, упаковка, тулинг

- **pytest + Hypothesis.** Детерминированное ядро → property-тесты на инварианты: HP не уходит в минус, пороги стресса монотонны (раздел 8.1), формула урона (7.2) корректна на границах. Аудитория найдёт каждый edge-case — лучше найти раньше.
- **Упаковка:** `pyproject.toml`, build-backend **Hatchling**, dev-окружение/локфайл через **uv**. Дистрибуция через `pipx install dungeon-clash` → команда `dungeon` в PATH. Это и есть «живёт в твоём терминале» (раздел 1).
- **Тулинг:** `ruff` (линт+формат), `mypy` (типизация `state`-API — публичный контракт, от которого зависят стратегии игроков), `pre-commit`, GitHub Actions CI (тесты + валидация контент-TOML против схемы).

## 9. Эволюция графики: ports-and-adapters

Опасение «заложимся на терминал → не сможем потом добавить графику» закрывается именно развязкой ядра и рендера. Графический клиент потребляет ровно то же, что и терминальный: **состояние + поток событий + контент**. Рендер — сменный адаптер к неизменному ядру.

```
Core (headless, детерминированный)
  ├─ state: GameState
  ├─ events: [HitEvent, StressEvent, ...]   ← семантика, не готовые строки
  └─ content: враги/реликвии/спрайты как данные (ID-ссылки на ассеты)

Renderer (порт):  render(state, events) -> вывод

Адаптеры (сменные, без изменений в ядре):
  • TerminalRenderer  (Textual/Rich)             ← v1
  • TerminalImage     (kitty graphics / sixel)    ← дешёвый апгрейд, тот же стек
  • DesktopRenderer   (pygame / Arcade / Godot)   ← переиспользует core+content
  • WebRenderer       (canvas/WebGL, кормится FastAPI) ← MMO-клиент
```

### Что переиспользуется, а что переписывается — честно

При добавлении графики переписывается **только view-слой** (Textual-виджеты по природе не переезжают в окно Godot — презентация платформо-зависима). Но **игра не переписывается**:

| Слой | При добавлении графики |
|------|------------------------|
| Ядро правил (бой, стресс, push-your-luck) | переиспользуется 1:1 |
| Контент (враги, реликвии, заклинания в TOML) | переиспользуется 1:1 |
| Исполнение стратегий игрока (Python) | переиспользуется 1:1 |
| Персистентность (SQLite) | переиспользуется 1:1 |
| **Рендер** | пишется новый |

Это лучший достижимый исход: не существует архитектуры, где GPU-графику добавляют без написания GPU-рендера. Вопрос лишь в том, тянется ли за этим переписывание логики — при развязке нет. Парадокс: взять тяжёлый движок (tcod/Godot) **сейчас** и размазать его вызовы по логике — это *бóльший* лок-ин, чем развязанный терминальный клиент.

### «Лучше графика» ≠ обязательно «уйти из терминала»

Современные терминалы (kitty, iTerm2, WezTerm) умеют настоящие картинки через kitty graphics protocol / sixel. Можно показывать PNG-спрайты прямо в терминале, сохранив весь стек — дешёвый промежуточный апгрейд без смены архитектуры.

### Три правила границы, которые закладываем с первого дня (стоят ≈ ничего)

1. **Ядро не импортирует Textual/Rich.** Зашиваем правилом в `import-linter` в CI — нарушение границы падает на сборке.
2. **Ассеты по стабильным ID** (`enemy_id`, а не инлайн-ASCII в логике). Графический клиент потом мапит тот же ID на спрайт вместо ASCII-арта.
3. **События несут семантику** (зона, урон, результат), а не готовые строки — чтобы графика могла их *анимировать*, а не просто печатать (см. §3, рендер-пайплайн).

Справедливая оговорка: view под любую новую графику придётся написать заново — этого не избежать в принципе. Но это ~20% работы; ядро, контент, стратегии и персистентность (вся ценность игры) переезжают без изменений.

## 10. ML-агенты и RL: ядро как обучающая среда

Требование «в игру могут играть ML-модели, и её можно использовать для RL, чтобы стратегии подбирались моделями» архитектуру не ломает — детерминированное headless-ядро, эмитящее события, **и есть RL-environment**. RL-обёртка — это третий адаптер к тому же ядру, в ряд с рендером (§9) и persistence.

### Ключевая идея: «стратегия-как-код» → «стратегия-как-policy»

Человек пишет `strategy(state) -> action`; RL-policy — `policy(obs) -> action`. Единый протокол делает ML-модель **просто ещё одним поставщиком стратегии** в тот же слот. Отдельного «RL-режима» не нужно:

```python
class Strategy(Protocol):
    def decide(self, state: GameState) -> Action: ...

# взаимозаменяемы:
#   HumanStrategy  — Python-функция игрока
#   ScriptedBot    — эвристика (тесты, болванчики)
#   RLPolicy       — обученная модель + encode(state) -> obs
```

Прямое расширение пиллара GDD (раздел 5): «strategy as code» → «strategy as policy».

### Gymnasium-обёртка поверх ядра

```
core.step(state, action, rng) -> (new_state, events)
        │  адаптер
        ▼
GymEnv.reset() -> obs
GymEnv.step(action) -> (obs, reward, terminated, truncated, info)
```

Достраивается вокруг **неизменного** ядра:

- **Observation space.** `encode(state) -> obs`: непрерывное (`hp_pct`, `stress_pct`) → float; дискретное (тип комнаты/врага, реликвии) → one-hot/эмбеддинги. `gym.spaces.Dict`/`Box`. Схема версионируется (контент растёт).
- **Action space.** Бой структурный: зона атаки (3) × зона защиты (3) × опц. заклинание × предмет → `MultiDiscrete`. Из-за условной доступности в GDD (Greatsword без LEGS, нет кристаллов — нет каста, действия открываются прогрессией) — **обязателен action masking** (`info["action_mask"]`, тренировка через MaskablePPO). Поднимает sample-efficiency, убирает невалидные действия.
- **Reward — подключаемый, из потока событий.** `reward = f(events, Δstate)`: sparse (выжил/золото на extract) или shaped (урон, очищенные этажи, контроль стресса). Тот же event-stream, что и для логов.
- **Seeding/детерминизм** — уже есть; Gymnasium-seeding ложится на per-run RNG, эпизоды воспроизводимы.
- **Варианты env**: combat-only (короткий горизонт) vs full-run (с навигацией/extract). Этажи GDD — готовый **curriculum**.

### Мульти-агентность: PettingZoo

Параллельно Gymnasium на том же ядре — для self-play, будущего PvP/MMO (v3) и многовражеских боёв (Swarm как агенты). Self-play — естественный путь «модели подбирают стратегии друг против друга».

### Два смысла «стратегии подбираются моделями»

1. **RL над пространством действий** — policy-сеть учится играть (SB3/CleanRL/RLlib).
2. **Поиск по пространству Python-кода стратегий** — уникально возможен из-за «strategy as code»: LLM/эволюция генерируют код `strategy(state)`, симулятор служит fitness-функцией, лучшие отбираются.

Оба используют одно ядро как «зал испытаний».

### Пропускная способность — единственное реальное напряжение

RL жрёт 10⁶–10⁸ шагов. Стратегия — тиерами, без преждевременного переписывания:

| Тиер | Подход | Когда |
|------|--------|-------|
| 1 | Чистый Python + векторизация (`AsyncVectorEnv`/Subproc по ядрам CPU) | старт. Игра пошаговая, obs низкоразмерный (не пиксели) — Python часто достаточен |
| 2 | Numba/Cython на горячих путях боя | если шаг стал бутылочным горлышком |
| 3 | Компилируемое ядро (Rust/C) **за тем же интерфейсом** или JAX-ядро (gymnax-стиль) на GPU | только для research-масштаба |

И человеческие Python-стратегии, и детерминированный replay (анти-чит) сохраняются на всех тиерах — ускоряем *движок симуляции*, а не контракт стратегии. JAX наперёд не берём: ограничения jit дорого стоят и расходятся с человеко-играбельным ядром.

### Граница зависимостей

Игра шипает **лёгкий env-пакет** (Gymnasium/PettingZoo + NumPy-энкодер). Тренеры (Stable-Baselines3, CleanRL, RLlib, gymnax) — **опциональные extras**, не зависимости ядра. Researcher приносит свой trainer; мы даём среду.

### Что заложить с первого дня (≈бесплатно)

- единый `Strategy`-протокол `decide(state) -> action`;
- `encode(state) -> obs` и `action_mask(state)` в адаптерном слое, **не** в ядре;
- reward как подключаемая функция от событий;
- версионирование obs-схемы.

## 11. MMO-слой (v3/v4): спланировать сейчас, не строить сейчас

- **Сервер:** **FastAPI** — async, Pydantic-native, переиспользуем те же модели `state`/контента, что и клиент.
- **БД:** **PostgreSQL** — общий персистентный мир, гильдии, лидерборды.
- **Анти-чит:** клиент шлёт seed + стратегию; сервер **реиграет прогон тем же детерминированным ядром** и проверяет результат. Лидерборды «по эффективности стратегий» (v3) без этого невозможны.
- **Async co-op** («видеть героев других игроков в своём логе»): event-feed, polling или websockets.
- **Market data (v4):** плановая задача дёргает рыночное API → модификатор сложности. Тривиальный cron на сервере.
- **Контент-сообщества:** GitHub PR-флоу + слой голосования; CI валидирует против Pydantic-схемы (см. §5).

Ничего из этого не строим в v1 — но детерминированное ядро (§3) и контент-как-данные (§5) делают этот путь дешёвым, если заложить их сразу.

## 12. Что я сознательно НЕ выбираю

| Анти-выбор | Причина |
|------------|---------|
| Переписать ядро на Go/Rust/C | Убивает пиллар «стратегия-как-код на Python». |
| Web/Electron-фронтенд | Нарушает «терминал-нативность, уважает внимание» (раздел 1, 2). |
| Игровой движок (Godot/Unity) сейчас | Неверная парадигма для ASCII/TUI; и это *бóльший* лок-ин, чем развязанный клиент (§9). Дверь к графике держим открытой через ports-and-adapters, а не через выбор тяжёлого движка наперёд. |

| `curses` напрямую | Слишком низкоуровнево; Textual/Rich дают тот же результат на порядок дешевле. |

## 13. Карта «требование GDD → технология»

| Требование GDD | Решение в стеке |
|----------------|-----------------|
| Стратегия-как-код (раздел 5) | Python-ядро, in-process исполнение, frozen `state` (Pydantic) |
| Активный режим, event-driven (4.2) | Textual |
| `status` / `log` / `--stats` (4.1, 4.2) | Typer + Rich + SQLite-агрегаты |
| Никогда не наказывать за отсутствие (4.3) | Ленивая offline-симуляция + seeded RNG |
| Детальный лог каждой смерти (4.2) | Детерминированное ядро + append-only лог в SQLite |
| Мета-прогресс не теряется (15, 10.3) | Нормализованные таблицы SQLite |
| Тяжёлый контент + PR-контент (11–14, v4) | TOML + Pydantic + CI-валидация |
| Лидерборды по стратегиям (v3) | Серверный replay тем же ядром (FastAPI + Postgres) |
| Sharing/исполнение чужих стратегий (v2/v3) | Песочница: WASM/subprocess |
| «Живёт в терминале» (1) | pipx-устанавливаемый пакет, команда `dungeon` |
| Опция будущей графики (десктоп/web/картинки в терминале) | Ports-and-adapters: сменный рендер поверх неизменного ядра (§9) |
| ML-модели играют в игру / RL-обучение стратегий | Gymnasium/PettingZoo-адаптер поверх ядра, единый `Strategy`-протокол, action masking (§10) |
